## B. 接口改造清单

## B1. P0（必须先做）

1. 排行接口区分运行类型
- 接口：
  - `GET /api/ranking?target_date=YYYY-MM-DD&run_type=prod`
- 要求：
  - 默认 `run_type=prod`
  - 前端只读 `prod` 结果

2. 评分接口支持“发布/试算”
- 接口：
  - `POST /api/scoring/run`
- Body：
```json
{
  "target_date": "2026-02-24",
  "publish": true,
  "run_type": "prod"
}
```
- 规则：
  - `publish=true` 才更新生产排行
  - `publish=false` 只产出草稿

3. 回测接口返回 `run_id`
- 接口：
  - `POST /api/backtest/run?days=60`
- 返回：
```json
{
  "status": "processing",
  "run_id": "bt_20260224_001"
}
```
- 目的：
  - 后续查询任务状态、明细

4. AI 确认接口幂等
- 接口：
  - `POST /api/ai/confirm`
- 返回应包含：
```json
{
  "status": "ok",
  "added": 12,
  "skipped": 3
}
```

5. 配置接口避免覆盖 API Key
- 接口：
  - `GET /api/config` 返回 `api_key_set/api_key_masked`
  - `POST /api/config` 空 key 不更新

## D. 开发顺序（推荐）

1. 先做回测任务状态机一致性回归（创建/取消/重试/终态）。  
2. 再做 `run_type=prod/backtest` 隔离回归（同日结果不互相污染）。  
3. 再做回测结果口径对齐（`/results` 与 `/results/{date}`）。  
4. 最后补齐本阶段验收脚本与文档。  

---

## E. 验收清单（最小）

1. `POST /api/backtest/run` 返回 `run_id`，且可在 `/api/backtest/jobs` 查到。  
2. 回测任务状态能正确流转到 `completed/failed/cancelled` 终态。  
3. `cancel/retry` 仅在允许状态生效，返回信息可解释。  
4. `run_type=prod` 与 `run_type=backtest` 查询结果可区分，生产排行不被回测污染。  
5. 回测趋势与单日明细接口返回一致口径数据。  
6. 质量门控回归脚本持续通过（作为本阶段回归基线）。
